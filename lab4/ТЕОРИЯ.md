# Теория: резервное копирование и восстановление в Microsoft SQL Server

## 1. Модели восстановления (recovery model)

Модель восстановления базы определяет, как ведётся журнал транзакций и какие типы бэкапов возможны.

- **SIMPLE** — журнал транзакций минимален; после checkpoint пространство журнала переиспользуется. Нельзя делать резервные копии журнала (BACKUP LOG). Восстановление возможно только до момента последнего полного или дифференциального бэкапа.
- **FULL** — все операции логируются. Можно выполнять BACKUP LOG; возможна цепочка лог-бэкапов и восстановление к произвольному моменту времени (point-in-time restore).
- **BULK_LOGGED** — массовые операции частично не логируются детально; компромисс между FULL и SIMPLE для больших вставок/индексов.

Для построения цепочки Full → Log → Diff и восстановления к моменту времени база должна быть в режиме **FULL**.

## 2. Типы резервных копий

### 2.1. Полная резервная копия (FULL backup)

- Копирует **всю** базу данных на момент выполнения: все данные, объекты, часть журнала, необходимая для согласованного состояния.
- Является **базой** цепочки восстановления: все последующие дифференциальные и лог-бэкапы опираются на последний полный бэкап.
- Параметры: `BACKUP DATABASE ... TO DISK = ... WITH INIT, COMPRESSION, STATS = 10`. `COMPRESSION` уменьшает размер и время копирования.

### 2.2. Дифференциальная резервная копия (DIFFERENTIAL backup)

- Копирует только **изменённые** данные с момента **последнего полного** бэкапа (изменённые экстенты).
- Каждый следующий дифференциальный бэкап включает все изменения с момента того же полного бэкапа (т.е. diff2 содержит diff1 + новые изменения).
- При восстановлении достаточно применить **один** полный бэкап и **последний** дифференциальный (не обязательно все подряд). Для учебной демонстрации цепочки часто применяют diff1 → diff2 → diff3 по порядку с NORECOVERY.
- Параметры: `BACKUP DATABASE ... TO DISK = ... WITH DIFFERENTIAL, INIT, COMPRESSION`.

### 2.3. Резервная копия журнала транзакций (LOG backup)

- Копирует **журнал транзакций** с момента последнего лог-бэкапа (или с момента полного бэкапа, если лог-бэкапов ещё не было).
- Возможна только при модели восстановления **FULL** (или BULK_LOGGED).
- Непрерывная цепочка лог-бэкапов позволяет восстановить базу к **любому моменту времени** между бэкапами (point-in-time restore).
- Параметры: `BACKUP LOG ... TO DISK = ... WITH INIT, COMPRESSION`.

## 3. Цепочка восстановления (backup chain)

Логическая последовательность:

1. **Полный бэкап** — база (и при FULL — начало цепочки журнала).
2. **Опционально:** один или несколько **лог-бэкапов** — непрерывная последовательность записей журнала.
3. **Опционально:** один или несколько **дифференциальных бэкапов** — изменения с момента последнего полного.
4. **Опционально:** снова **лог-бэкапы** после дифференциальных — для восстановления к моменту времени после diff.

При восстановлении порядок применения:

- Восстановить **полный** бэкап с `NORECOVERY`.
- Применить **дифференциальные** по порядку (каждый с `NORECOVERY`), последний при необходимости с `RECOVERY`, либо после последнего diff применить **лог-бэкапы** с `NORECOVERY`, последний — с `RECOVERY` или с `STOPAT` для point-in-time.
- Если нужен момент времени внутри лога — применить лог-бэкапы с `STOPAT = 'дата_время'`.

**NORECOVERY** — база остаётся в состоянии «восстановление», можно применять следующие бэкапы.  
**RECOVERY** — завершение восстановления, база переводится в обычное состояние, дальнейшие бэкапы к этой цепочке применить нельзя.

## 4. Восстановление на другом экземпляре (второй сервер)

- Файлы данных и журнала на другом сервере обычно имеют другие пути. Поэтому при первом `RESTORE DATABASE` из полного бэкапа нужно указать **MOVE**: логическое имя файла → новый физический путь.
- Логические имена файлов берутся из бэкапа (их можно посмотреть: `RESTORE FILELISTONLY FROM DISK = '...'`).
- Пример: при восстановлении из бэкапа базы, созданной на первом сервере с файлами `ProjectDB_Lab4.mdf` и `ProjectDB_Lab4_log.ldf`, на втором сервере логические имена могут остаться теми же, что у исходной базы (например `ProjectDB_Lab2` и `ProjectDB_Lab2_log`), а физические пути задаются так:  
  `MOVE N'ProjectDB_Lab2' TO N'/var/opt/mssql/data/ProjectDB_Lab4_sql2.mdf'`,  
  `MOVE N'ProjectDB_Lab2_log' TO N'/var/opt/mssql/data/ProjectDB_Lab4_sql2_log.ldf'`.
- Если база с таким именем уже существует и нужно перезаписать — используется **REPLACE** в первом RESTORE.

## 5. Восстановление к моменту времени (point-in-time restore)

- Возможно только при модели **FULL** и при наличии **непрерывной** цепочки лог-бэкапов от полного (или последнего применённого дифференциального) бэкапа до момента, к которому восстанавливаем.
- Команда: `RESTORE LOG ... FROM DISK = ... WITH NORECOVERY, STOPAT = 'YYYY-MM-DD HH:MM:SS'`, затем последний RESTORE LOG с `RECOVERY` (или с `STOPAT` и `RECOVERY`).
- Если в цепочке лог-бэкапов есть пропуск (например, не сделан один из лог-бэкапов), восстановление к моменту времени после пропуска невозможно — согласованность данных не гарантируется.

---

## 6. Системная база master

- **master** хранит системные объекты экземпляра: логины, настройки сервера, список баз данных и т.д. Экземпляр SQL Server при запуске читает **master** и от него зависит.
- **Резервное копирование master** выполняется как обычный полный бэкап: `BACKUP DATABASE master TO DISK = '...' WITH COMPRESSION`.
- **Восстановление master** возможно только когда экземпляр запущен в режиме **single-user** (`-m`), потому что в обычном режиме с **master** работают системные процессы и другие соединения. В single-user допускается только одно подключение (например, sqlcmd), которое выполняет `RESTORE DATABASE master FROM DISK = '...' WITH REPLACE`.
- После успешного **RESTORE DATABASE master** экземпляр **перезапускается**: соединение обрывается, процесс sqlservr перезапускается и заново читает восстановленную копию **master**. Поэтому восстановление master всегда сопровождается перезапуском службы/контейнера.

## 7. Контрольные вопросы (ответы кратко)

1. **Чем отличаются FULL, DIFFERENTIAL и LOG backups и как они связаны в цепочке восстановления?**  
   FULL — полная копия базы; DIFFERENTIAL — изменения с момента последнего FULL; LOG — последовательность записей журнала. В цепочке сначала применяется FULL, затем при необходимости один или несколько DIFF, затем при необходимости LOG-бэкапы до нужного момента. Каждый следующий бэкап опирается на предыдущие.

2. **Почему для point-in-time restore важна непрерывная цепочка log backups?**  
   Журнал транзакций фиксирует все изменения в строгом порядке. Пропуск в цепочке лог-бэкапов означает потерю части изменений; восстановить согласованное состояние на произвольный момент времени после пропуска нельзя.

3. **Почему восстановление master требует single-user mode и приводит к перезапуску экземпляра?**  
   **master** используется самим сервером при работе; в обычном режиме его нельзя перезаписать при активных соединениях. В single-user допускается одно подключение для выполнения RESTORE. После восстановления **master** сервер должен перечитать обновлённую системную базу, поэтому процесс sqlservr перезапускается, что разрывает все соединения.
