# Модели восстановления (recovery model)

## Теория

Модель восстановления базы данных в SQL Server определяет, как ведётся журнал транзакций и какие типы резервных копий допустимы.

- **SIMPLE** — журнал транзакций после checkpoint переиспользуется; операция `BACKUP LOG` недоступна. Восстановить базу можно только до момента последнего полного или дифференциального бэкапа.
- **FULL** — все операции логируются; можно выполнять резервное копирование журнала и строить непрерывную цепочку лог-бэкапов для восстановления к произвольному моменту времени (point-in-time restore).
- **BULK_LOGGED** — компромисс: массовые операции логируются не полностью; лог-бэкапы возможны, но восстановление к моменту времени внутри массовой операции ограничено.

Для лабораторной работы сначала используется **SIMPLE** (исходная база и первый полный бэкап), затем целевая база переводится в **FULL** для построения цепочки full → log → diff.

---

## Как это сделано в лабораторной

Исходную базу перед полным бэкапом переводим в SIMPLE:

```sql
USE master;
GO

ALTER DATABASE [ProjectDB_Lab2] SET RECOVERY SIMPLE WITH NO_WAIT;
GO
```

После создания новой базы ProjectDB_Lab4 из этого бэкапа переводим её в FULL для дальнейших лог- и дифференциальных бэкапов:

```sql
USE master;
GO

ALTER DATABASE [ProjectDB_Lab4] SET RECOVERY FULL WITH NO_WAIT;
GO
```

Дальнейшие полный бэкап, лог-бэкап и три дифференциальных выполняются уже при модели FULL.
